Joanie Abalone <pfenerty at earthlink.net> wrote:

> the problem that i see is that cdrecord -pad successfully pads a file that it
> thinks is of length 79499220.
>
> but the file is actually much larger:
>
> -rw-rw-r-- 1 pf pf 127866524 Apr 21 10:16 hayes_walkonby_latest.wav
>
> i have no idea what process decided that the file size is 79499220, or why
> that process consistently gets only this one filesize wrong, among hundreds
> (thousands?) of files handled correctly ... only that cdrecord complains
> about it.
>
> however, it's probably quite telling that cdrecord correctly writes out the
> entire file when /usr/bin/normalize is not invoked.

Well, then it looks like some program was called that introduced a bad *.WAV
header to that file. Cdrecord reads the WAV header and uses the size that is in 
the WAV header unless the file is actually shorter than announced by the WAV 
header.
