Hello everyone,

Using mkisofs 3.0.0 on Slackware64-Current with k3b 2.0.1 I have 
corrupted files on the resulting image file.

Please see my bug report on k3b here for more details if possible 
https://bugs.kde.org/show_bug.cgi?id=257241

To summarize it I've gone down to the "-sort" option using the two 
command lines below :
Non-working :

/usr/bin/mkisofs -gui -graft-points -volid "Vol 1" -volset "" -appid "K3B THE
CD KREATOR (C) 1998-2010 SEBASTIAN TRUEG AND MICHAL MALEK" -publisher ""
-preparer "" -sysid "LINUX" -volset-size 1 -volset-seqno 1 -sort k3bmj3024.tmp
-rational-rock -hide-list k3bkL3024.tmp -no-cache-inodes -udf
-full-iso9660-filenames -iso-level 3 -path-list k3beo3024.tmp

Working :
/usr/bin/mkisofs -gui -graft-points -volid "Vol 1" -volset "" -appid "K3B THE
CD KREATOR (C) 1998-2010 SEBASTIAN TRUEG AND MICHAL MALEK" -publisher ""
-preparer "" -sysid "LINUX" -volset-size 1 -volset-seqno 1 -rational-rock
-hide-list k3bkL3024.tmp -no-cache-inodes -udf -full-iso9660-filenames
-iso-level 3 -path-list k3beo3024.tmp


The result on the non working command line is that all files < 4GB are 
corrupted which seems very strange to me.
I would have understood more easily that small files would be ok and big 
ones corrupted but it's the opposite here...

I've spent time tracking down which option does this (-sort, and by the 
way the file passed to it is empty here) but didn't watch out "how" the 
files were corrupted. I can extract the TOC or other parts of both 
working and corrupted ISO results if needed to reproduce the bug.
You can also reproduce it easily by creating a project mixing small 
(<4G) and big (>4G) files using the above options on mkisofs 3.0.0, 
especially -sort with an empty file as argument.

I didn't see any documentation to how "officially" report bugs so I'm 
using the list.I hope it's ok.

Thanks,
Best regards,
Stephane
