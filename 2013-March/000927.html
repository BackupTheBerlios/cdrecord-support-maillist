By using an undocumented dev= parameter, you force cdrecord to use a specific 
driver on a Linux system that supports more than one driver per device.

If you use the official documented dev= syntax, cdrecord choses the best 
available driver.

An early abort may be a result of trying to use media that is not supported by 
your drive (try cdrecord -atip, cdrecord -minfo -v to get the media 
manufacturer and state).

In addition, on Linux there are hostile programs that interrupt the write 
process. These programs are "hald" or it's successors (all written by the same 
person who is not interested in understanding drive states for optical media).

If you use chean unreliable media, try to use Verbatim media that is identified 
as "Verbatim" by cdrecord -atip.
