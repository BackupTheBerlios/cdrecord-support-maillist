> If I eject and the retract the tray and try the mount again it works fine.

This seems to be a known issue under Linux:

http://lkml.indiana.edu/hypermail/linux/kernel/0705.0/0259.html

even as far back as 1996:

http://lkml.indiana.edu/hypermail/linux/kernel/9604.3/0377.html

which is a more serious annoyance with laptop drives that cannot
close their trays without the help of a human hand.

It seems the problem was added to a list of kernel regressions:

http://lkml.indiana.edu/hypermail/linux/kernel/0705.0/0278.html

(at very bottom of post) but seems to be no longer present, so
newer kernels may not suffer from it. If so, I'd like to know
when the change was made.

What I would like, and suspect it exists, is some command that
causes the kernel to invalidate all cache information and forces
a total "media reevaluation" of a drive:

driveresetcmd /dev/dvd

If it can do it as part of an eject, it should be able to do
it without actually ejecting it.

I did a search for 

linux invalidate read cache

but have not yet found the magic chant. 

One possibility (at least for ATA drives) is

hdparm -f /dev/dvd

but on my older system (haven't tried it on newer machines) this
results in

"BLKFLSBUF failed: Function not implemented".

Other possibilities include things with names like flush, fsync,
sync, etc.
