"Joshua Newton" <cdrecord at standingduck.net> wrote:

> cdrecord seems to be very confused by SATAPI DVD burners in Linux. 
> Using the ATA: interface causes cdrecord to mis-detect the state of DMA
> (to be fair, so does hdparm, but qpxtools gets it right), and it refuses
> to burn at any speed higher than 16x, and quits.

???

> Passing driveropts=burnfree causes cdrecord to make the same complaint,
> but it continues to the burning stage, claiming that it's beginning to
> burn at 24x, but cdrecord reports an actual speed of 16-17x through the
> entire burn (a full-size 700MB ISO image).  Is cdrecord written to
> libata yet?  Is there an undocumented switch along the lines of "-sata" 
> or "-forcespeed"?  (I know about forcespeed, but that appears to be a
> method of telling cdrecord to override the drive; not its own DMA
> detection routine.)

This looks confused. Platform specific interfaces in the kernel are irrelevent 
for generic SCSI. Cdrecord uses documented interfaces that are available to
the user space commands and I encourage Kernel hackers to implement _one_
orthogonal interface for all SCSI transfers.

> Having cdrecord outguess the user seems suboptimal.
>
> (FTR: Plextor PX-716SA, v1.11 f/w)

This drive is known to support no DMA for the READ buffer command.

You need to use the documented workarounds for such drives, e.g. setting the 
environment CDR_FORCESPEED
