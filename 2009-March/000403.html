Sigh.
The file would just have the path spec that you otherwise would put on the
command line.
Something like this: myfiles.txt
/usr/local/file1=/usr/local/file1
/usr/file2=/usr/file2
...

And the command line would have -graft-points -path-list myfiles.txt

> I didn't know one could list *files*
> within the -path-list file. I assume these default to the root
> directory of the image. Do you mean that one can use -path-list
> to include a large number of files (and contents of directories) to
> the root directory of the image without exceeding the maximum command
> line length, however, to map files to arbitrary paths on the
> image one has to use -graft-points and thus the command line limit
> becomes an issue. If so, there still is a need for a -graft-points-list.


-graft-points changes how you can specify the path spec.
-path-list allows you to have the path spec in a file. Combine both of them, and
you get your "-graft-points-list".
It is _that_ simple.
